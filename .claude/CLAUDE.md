# my_ui — 项目规范

> 个人 UI 设计作品集。主站用 Next.js，部署在 Vercel。每个作品是独立的 UI 演示，同时被主站聚合展示。

---

## 项目目标

1. **对外展示**：主站是一个可访问的网站，展示所有 UI 作品。首页以缩略图/卡片形式呈现所有素材，点击进入后可查看该素材。
2. **素材复用**：作品中沉淀的组件、动效、布局、颜色系统可以被抽取，用于未来的新作品或外部项目。
3. **独立运行**：每个作品可以脱离主站独立运行，直接在浏览器中打开即可工作。

---

### 关键目录说明

- **`app/`** — Next.js 主站。首页是素材网格，点击素材进入详情页。详情页目前只需能正常进入即可，渲染方式后续确定。
- **`registry/`** — 素材库。每个作品是一个独立模块，可以脱离主站运行。编号：`01`, `02`, `03`...（两位数）。
- **`data/works.ts`** — 素材注册表。所有作品的元数据集中在这里，主站从这里读取。
- **`styles/tokens.css`** — Design Tokens，跨所有作品共享。

### 素材集成方式（后续确定）

主站目前只需要实现路由入口：首页点击素材卡片 → 进入详情页。详情页如何渲染该素材（iframe、独立页面、React 组件封装等）后续根据素材复杂度逐步确定。

素材类型差异：
- 简单素材：纯 HTML/CSS/JS，可能通过 iframe 或静态托管直接展示。
- 复杂素材：涉及 Canvas、WebGL、第三方库，渲染方式需单独处理。

## 文件命名规范

### 作品文件夹

`NN-slug/`，如 `01-field`、`02-night`、`03-ocean`。两位数编号，留足扩展空间。

### 作品内部文件（当前阶段）

每个作品至少包含：

| 文件名 | 用途 |
|--------|------|
| `meta.ts` | 作品元数据 |
| `design.md` | 设计文档 |
| `index.html` | 独立运行入口 |
| `assets/` | 图片、字体、视频等素材 |

> 文件结构后续根据素材复杂度调整。简单素材可能只有 HTML/CSS/JS；复杂素材可能包含 Canvas、WebGL、构建产物或多个入口文件。

### 素材文件前缀

| 前缀 | 用途 |
|------|------|
| `01-` | 原始参考图 |
| `02-` | Web 背景图 |
| `03-` | UI 概念图 / Mockup |
| `04-` | 移动端素材 |

## 开发工作流

### Commit 频率

**任何修改后必须立即 commit。** 包括代码、样式、素材、文案、文档。不允许积攒多个修改再一次性提交。

### Commit 规范

```
<类型>: <中文描述>

<可选的详细说明>
```

类型：

| 类型 | 用途 |
|------|------|
| `feat` | 新功能、新作品、新组件 |
| `fix` | 修复问题 |
| `style` | 样式调整 |
| `refactor` | 重构，不改变功能 |
| `docs` | 文档更新 |
| `asset` | 素材更新 |
| `chore` | 配置、脚本等杂项 |

### 推送规则

**每次 commit 后立即 push。**

```bash
git add .
git commit -m "type: 描述"
git push origin main
```

### 分支策略

- `main` 是唯一分支，始终可运行、可展示。
- 不创建 feature 分支，所有开发在 main 上进行。

## 迭代开发模式

> 本项目采用**持续迭代**的开发方式。完成一个目标后，不停止，而是以资深用户视角自我审查，提出新需求，进入下一轮迭代。

### 迭代循环

```
实现功能 → 自我审查 → 提出改进需求 → 规划改进方案 → 实现改进 → 验证 → 循环
```

### 1. 自我审查（资深用户视角）

每完成一个功能或阶段后，必须进行自我审查。审查维度包括：

- **视觉完成度**：是否符合 DESIGN.md 的设计规范？颜色、间距、字体是否一致？
- **交互体验**：动画是否流畅？交互反馈是否明确？是否存在卡顿或闪烁？
- **代码质量**：是否有重复代码？是否有更好的实现方式？类型定义是否完整？
- **响应式**：在移动端和桌面端是否都有良好的体验？
- **可访问性**：键盘导航是否正常？是否有适当的 ARIA 标注？
- **性能**：是否存在不必要的重渲染？图片是否优化？
- **边界情况**：空状态、加载状态、错误状态是否处理？

审查结果必须形成具体的改进需求列表，写入当前工作的 TODO 或备注中。

### 2. Agent Team 协作（推荐）

对于复杂任务，推荐使用多 Agent 协作模式：

- **实现 Agent**：负责具体的代码实现。
- **审查 Agent**：负责代码审查、提出改进建议。
- **测试 Agent**：负责验证功能是否按预期工作，检查边界情况。
- **设计 Agent**：负责视觉细节调整，确保符合 DESIGN.md。

多个 Agent 可以并行工作，但最终所有产出必须**收敛到同一套代码和文档**中。

### 3. 收敛与检验

无论使用多少 Agent，最终必须：

- **代码收敛**：所有修改合并到同一分支（main），确保没有冲突。
- **文档收敛**：所有设计决策、API 变更、新增组件必须在文档中体现。
- **验证收敛**：所有功能必须通过验证（手动检查或运行测试），确保可运行、可展示。
- **一致性收敛**：所有产出必须符合 DESIGN.md 的设计规范和 CONSTRAINTS.md 的技术约束。

### 4. 持续迭代

收敛完成后，回到步骤 1（自我审查），提出下一轮改进需求。迭代永不停止，直到用户明确喊停。

### 迭代记录

每次迭代结束后，在维护记录中追加一条记录，说明：
- 本轮实现了什么
- 发现了什么问题
- 下一轮要改进什么
